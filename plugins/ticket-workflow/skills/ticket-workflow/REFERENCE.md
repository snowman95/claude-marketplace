# ticket-workflow — Phase 상세 절차

## Phase 0: 컨텍스트 로딩

어느 모듈/화면이 대상인지 불명확하면 사용자에게 확인한다 (추정으로 잡지 않음).

1. `architecture.md` 로딩. 없으면 코드베이스 분석 후 새로 작성.
2. 해당 `modules/{모듈명}_research.md` 로딩. 없으면 Phase 1에서 생성.
3. 이미 충분한 컨텍스트가 있으면 Phase 1을 짧게 하거나 건너뛸 수 있다.

---

## Phase 1: Research

1. 관련 코드를 **깊이** 읽는다. 함수 흐름·의존성·엣지케이스까지 파악.
2. 발견 내용을 기존 문서에 반영:
   - 프로젝트 전체 구조 → `architecture.md`에 추가
   - 모듈 단위 분석 → `modules/{모듈명}_research.md`에 추가 또는 신규 생성
   - **티켓 모드**: 티켓 md에는 `## Phase 1: Research (요약)` 만 (불릿 요약 + `[[{모듈명}_research]]` 링크)
3. 기존 문서가 충분하면 짧게 끝낸다. 이미 아는 것을 반복 조사하지 않는다.

**완료 멘트**: `research 완료, 검토해주세요`
사용자 승인 후 Phase 2.

---

## Phase 2: Plan

1. research 승인 후에만 진행.
2. plan 작성 위치:
   - **티켓 모드**: `{JIRA-KEY}.md` 내 `## Plan` 섹션
   - **범용 모드**: `tasks/{모듈명}/{작업명}_plan.md` 신규 생성
3. plan에는 architecture.md·module research를 위키링크로 참조:
   ```
   > 참조: [[architecture]], [[{모듈명}_research]]
   ```
4. plan 포함 내용:
   - **목표**: 이 작업으로 달성하려는 것
   - **구현 단계**: 구체적 변경 내용, 코드 스니펫 포함
   - **수정 파일 경로**: 실제 변경될 파일 목록
   - **트레이드오프**: 고려한 대안과 선택 이유
   - **완료 기준**: 기능 동작, 테스트, 타입체크 통과 여부

**완료 멘트**: `plan 작성 완료, 검토해주세요` (구현 안 함)

---

## Phase 3: Annotation Cycle (주석 사이클)

1. 사용자가 Obsidian에서 plan에 인라인 메모를 추가한다.
2. "메모 처리해라" 지시를 받으면:
   - plan을 다시 읽고 모든 메모를 반영해 문서를 업데이트한다.
   - 아직 구현하지 않는다.
3. 사용자가 승인할 때까지 반복.

---

## Phase 4: Todo List

plan 하단에 세분화된 `## 구현 체크리스트`를 추가한다.
구현 중 진행 상황 추적기로 사용된다.

사용자 승인 후 Phase 5. 구현은 **「구현해라」** 등 명시 시.

---

## Phase 5: Implementation

"구현해라" 지시를 받으면:

- plan의 모든 항목을 실행한다. 선택적으로 하지 않는다.
- 단계를 끝낼 때마다 plan의 todo를 완료 표시한다.
- 모든 작업이 완료될 때까지 중간에 확인을 위해 멈추지 않는다.
- 구현 중 프로젝트 구조 발견이 있으면 `architecture.md` 또는 module research에 반영한다.
- **앱 소스** 편집은 이 단계에서만 (또는 사용자가 명시적으로 일괄 구현을 허용한 경우).
- **권장**: 코드 생성·다수 파일 수정은 서브에이전트에 위임. 가능하면 백그라운드로, 완료 시 결과 일괄 보고.

### 구현 후 (티켓 모드)

`tasks/**/{JIRA-KEY}.md` frontmatter 갱신.
